01 Tổng quan hệ thống

Hệ thống Enterprise AI Platform là một API Gateway thông minh đứng giữa người dùng và các nhà cung cấp AI (OpenAI, Google Gemini, Ollama, v.v.). Thay vì gọi trực tiếp đến từng nhà cung cấp, mọi yêu cầu đều đi qua hệ thống này để được kiểm duyệt, tối ưu, định tuyến, và ghi nhật ký.

💡 Ví dụ dễ hiểu

Hãy tưởng tượng hệ thống này giống như một tổng đài viên thông minh. Khi nhân viên muốn hỏi AI, thay vì gọi thẳng cho AI, họ gọi cho tổng đài. Tổng đài sẽ: (1) kiểm tra danh tính, (2) lọc thông tin nhạy cảm, (3) chọn AI nào phù hợp nhất, (4) chuyển cuộc gọi, (5) ghi lại chi phí.

Kiến trúc tổng thể

🧑 Người dùng (Trình duyệt) → 🖥️ Control Plane (Next.js :3000) → ⚡ API Gateway (Fastify :8080) → 🤖 Nhà cung cấp AI

📂 File: docker-compose.yml

Chỉ cần chạy docker compose up -d là cả 2 dịch vụ tự khởi động. Dữ liệu được lưu vĩnh viễn nhờ volumes (postgres_data, redis_data), nên tắt Docker không mất data.

services:
  postgres:                          # Cơ sở dữ liệu chính
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: postgres        # Tài khoản đăng nhập DB
      POSTGRES_PASSWORD: password    # Mật khẩu DB
      POSTGRES_DB: enterprise_ai     # Tên database

      - "5432:5432"                  # Cổng truy cập từ bên ngoài

  redis:                             # Bộ nhớ đệm tốc độ cao
    image: redis:7-alpine
    ports:
      - "6379:6379"
03 Xác thực & Bảo mật

Mọi request gửi đến Gateway đều phải qua 3 lớp bảo mật:

Lớp 1: JWT Token (Thẻ ra vào)

Khi người dùng đăng nhập thành công, hệ thống tạo ra một chuỗi mã hóa gọi là JWT Token. Token này giống như "thẻ ra vào tòa nhà" — mỗi lần gửi yêu cầu, người dùng phải đính kèm token này.

packages/auth/src/index.ts
// Khi nhận được request, hệ thống:
1. Lấy token từ header "Authorization: Bearer <token>"
2. Giải mã token bằng khóa bí mật (JWT_SECRET)
3. Kiểm tra session trong database (chưa hết hạn? chưa bị thu hồi?)
4. Gắn thông tin user vào request để các bước sau dùng:
   → request.user = { id, role, organizationId }

Lớp 2: Mã hóa API Key nhà cung cấp (AES-256-GCM)

Khi Admin nhập API Key của OpenAI/Gemini vào hệ thống, key đó KHÔNG lưu trực tiếp vào database. Thay vào đó, nó được mã hóa bằng thuật toán AES-256-GCM (chuẩn quân sự) rồi mới lưu.

packages/models/src/security/encryption.ts
// Mã hóa: "sk-abc123..." → "a1b2c3:d4e5f6:789xyz..."
//          (API Key gốc)     (dạng mã hóa: iv:authTag:encrypted)

encrypt("sk-abc123...")  // Khi Admin lưu API Key
decrypt("a1b2c3:...")    // Khi Gateway cần gọi AI, giải mã ra key gốc

Lớp 3: Mật khẩu người dùng (Argon2)

Mật khẩu người dùng được băm bằng Argon2 (thuật toán thắng cuộc thi PHC 2015). Ngay cả khi hacker lấy được database, cũng không thể khôi phục mật khẩu gốc.

04 Pipeline Chat chi tiết
⚡ Đây là phần quan trọng nhất!

Hiểu được luồng này = hiểu 80% toàn bộ hệ thống. File chủ chốt: apps/gateway/src/controllers/chat.ts

Khi người dùng bấm nút Send trên giao diện chat, đây là TOÀN BỘ những gì xảy ra bên trong:

Bước 1 — Frontend gửi request
Người dùng bấm Send → Frontend gọi API

File: apps/control-plane/src/lib/api.ts → hàm chatCompletionStream()

Frontend lấy toàn bộ lịch sử hội thoại + tin nhắn mới, đóng gói thành JSON và gửi đến Gateway qua endpoint POST /v1/chat/stream. Kèm theo JWT Token trong header.

// Dữ liệu gửi đi:
{
  "model": "gemini-2.0-flash",       // Model AI muốn dùng
  "messages": [                       // Lịch sử chat
    { "role": "user", "content": "Xin chào!" }
  ],
  "conversationId": "abc123"          // ID cuộc hội thoại (nếu có)
}
Bước 2 — Xác thực danh tính
Gateway kiểm tra JWT Token

File: packages/auth/src/index.ts → hook preValidation

Gateway nhận request, kiểm tra JWT Token. Nếu token hợp lệ → lấy ra userId và organizationId. Nếu không → trả về lỗi 401 Unauthorized.

Bước 3 — Quét an toàn (Safety Scan)
Kiểm tra PII + Prompt Injection

File: packages/safety/src/index.ts → class SafetyScanner

Hệ thống quét tin nhắn của người dùng qua 2 bộ lọc:

  • Lọc PII: Tìm và che dấu số điện thoại VN (09x, 03x...), số CCCD (12 chữ số), email → thay bằng [PHONE_REDACTED], [EMAIL_REDACTED]
  • Chống Prompt Injection: Phát hiện các câu lệnh nguy hiểm như "ignore all previous instructions", "you are now"... Nếu phát hiện → CHẶN ngay, không cho gửi đến AI.
Bước 4 — Kiểm tra chính sách (Policy Engine)
Áp dụng quy tắc của tổ chức

File: chat.ts dòng 114-137

Hệ thống tải tất cả Policy (quy tắc) đã được Admin thiết lập cho tổ chức, rồi kiểm tra từng quy tắc:

  • Ví dụ: "Nếu tin nhắn chứa PII → bắt buộc dùng model local (Ollama)" → chuyển strategy sang cost-optimized
  • Ví dụ: "Nếu tin nhắn chứa PII → từ chối yêu cầu" → ném lỗi ngay
Bước 5 — Kiểm tra Cache
Tìm câu trả lời đã lưu sẵn

File: packages/cache/src/index.ts → class SemanticCache

Trước khi gọi AI (tốn tiền + chậm), hệ thống tạo SHA-256 hash từ (model + messages) rồi tra trong Redis. Nếu trùng khớp → trả ngay kết quả cũ (tốc độ ~5ms, chi phí $0). Nếu không → tiếp tục bước 6.

Bước 6 — Định tuyến thông minh (Dynamic Router)
Chọn nhà cung cấp AI tốt nhất

File: packages/models/src/routing/DynamicRouter.ts

Đây là "bộ não" của hệ thống. Nó chấm điểm tất cả nhà cung cấp AI đã đăng ký dựa trên 7 tiêu chí, rồi chọn nhà cung cấp có điểm cao nhất. (Chi tiết ở Mục 05)

Bước 7 — Tạo Adapter & Gọi AI
Kết nối và gửi yêu cầu đến AI

File: packages/models/src/index.ts → class ModelFactory

Dựa trên nhà cung cấp được chọn, hệ thống tạo đúng loại Adapter (OpenAI, Ollama, Bedrock...) và gọi hàm chat() hoặc stream(). API Key được giải mã từ database ngay lúc này.

Bước 8 — Stream kết quả về Frontend
Trả từng chữ về cho người dùng (SSE)

File: chat.ts dòng 423-445

Gateway dùng Server-Sent Events (SSE) để gửi từng mảnh kết quả (chunk) về frontend theo thời gian thực. Giống ChatGPT — bạn thấy AI "gõ" từng chữ. Khi xong, gửi data: [DONE].

// Dữ liệu stream:
data: {"conversationId":"abc123"}     ← Chunk 1: ID hội thoại
data: {"text":"Xin"}                  ← Chunk 2: Từ đầu tiên
data: {"text":" chào"}                ← Chunk 3: Từ thứ hai
data: {"text":" bạn!"}               ← Chunk 4
data: [DONE]                          ← Kết thúc stream
Bước 9 — Lưu trữ & Ghi log
Ghi nhật ký, tính tiền, lưu cache

Sau khi stream xong, Gateway thực hiện song song:

  • Lưu tin nhắn vào bảng Message (PostgreSQL)
  • Ghi log vào bảng LogTrace (traceId, latency, cost, security flags)
  • Lưu cache kết quả vào Redis (để lần sau trả nhanh)
  • Tính chi phí qua CostTracker
  • Phát SSE cho dashboard admin (cập nhật real-time)
05 Hệ thống định tuyến AI (Routing)

Đây là hệ thống "chọn lựa thông minh" — quyết định mỗi request sẽ được gửi đến nhà cung cấp AI nào.

4 chiến lược định tuyến (Routing Strategy)

Chiến lượcƯu tiênKhi nào dùng
balancedCân bằng mọi yếu tốMặc định, phù hợp đa số trường hợp
cost-optimizedTiết kiệm chi phí tối đaKhi ngân sách hạn hẹp, ưu tiên Ollama (miễn phí)
latency-optimizedTốc độ phản hồi nhanh nhấtChatbot realtime, cần phản hồi tức thì
high-availabilityĐộ ổn định cao nhấtHệ thống quan trọng, không được lỗi

7 tiêu chí chấm điểm nhà cung cấp

File: packages/models/src/routing/ProviderScoreEngine.ts

  1. Health (Sức khỏe): Tỷ lệ lỗi gần đây. Lỗi nhiều = điểm thấp
  2. Availability (Khả dụng): Tỷ lệ hoạt động. Hay offline = điểm thấp
  3. Latency (Độ trễ): Thời gian phản hồi trung bình. Chậm = điểm thấp
  4. Cost (Chi phí): Giá tiền mỗi 1000 token. Đắt = điểm thấp
  5. Capability (Năng lực): Hỗ trợ streaming? Vision? Tool calling?
  6. Compliance (Tuân thủ): Có đáp ứng yêu cầu pháp lý không?
  7. Context Window: Cửa sổ ngữ cảnh có đủ lớn không?
Công thức tính điểm
Điểm tổng = (Health × w₁) + (Availability × w₂) + (Latency × w₃) 
          + (Cost × w₄) + (Capability × w₅) + (Compliance × w₆)
          + (ContextWindow × w₇) + (Priority × w₈)

// Trong đó w₁...w₈ là trọng số, thay đổi theo chiến lược.
// Ví dụ: cost-optimized → w₄ (Cost) = 3.0 (gấp 3 lần bình thường)

Circuit Breaker (Cầu dao tự ngắt)

File: packages/models/src/routing/CircuitBreaker.ts

Giống như cầu dao điện trong nhà. Khi một nhà cung cấp AI lỗi liên tục (ví dụ: OpenAI sập), Circuit Breaker sẽ tự động "ngắt mạch" — không gửi thêm request đến nhà cung cấp đó nữa mà chuyển sang nhà cung cấp khác.

🟢 CLOSED (Bình thường) → 🔴 OPEN (Ngắt mạch) → 🟡 HALF-OPEN (Thử lại) → 🟢 CLOSED

Failover (Tự chuyển đổi dự phòng)

Nếu nhà cung cấp AI #1 lỗi, hệ thống tự động thử nhà cung cấp AI #2, rồi #3 (tối đa 3 lần). Người dùng không biết đằng sau đang "chạy đua" như vậy.

06 Adapter — Kết nối với các AI

Mỗi nhà cung cấp AI có API riêng. Adapter là "bộ phiên dịch" giúp Gateway nói chuyện được với tất cả.

💡 Tại sao Gemini dùng OpenAIAdapter?

Google cung cấp một endpoint tương thích OpenAI tại https://generativelanguage.googleapis.com/v1beta/openai. Nhờ đó, hệ thống chỉ cần dùng thư viện OpenAI SDK nhưng trỏ đến URL của Google → giao tiếp bình thường.

ModelFactory — Nhà máy tạo Adapter

File: packages/models/src/index.ts

Khi cần gọi AI, hệ thống gọi ModelFactory.createAdapter(providerType, config). Hàm này nhìn vào providerType (ví dụ: "GOOGLE_GEMINI") rồi tạo đúng loại Adapter với đúng URL.

Bảng ánh xạ Provider → URL mặc định

GOOGLE_GEMINI → https://generativelanguage.googleapis.com/v1beta/openai
ANTHROPIC     → https://api.anthropic.com/v1
GROQ          → https://api.groq.com/openai/v1
OLLAMA        → http://127.0.0.1:11434
DEEPSEEK      → https://api.deepseek.com/v1
OPENROUTER    → https://openrouter.ai/api/v1
🐛 Bug đã sửa (quan trọng phải biết)

Trước đây, dòng code tạo adapter có bug: baseUrl: dbProvider?.baseUrl || process.env.OLLAMA_BASE_URL || 'http://127.0.0.1:11434'. Khi Provider không có baseUrl trong DB, nó fallback về Ollama URL cho MỌI provider (kể cả Gemini) → lỗi 404. Đã sửa thành chỉ fallback về Ollama URL khi providerType là 'OLLAMA'.

08 Cache thông minh (Semantic Cache)

File: packages/cache/src/index.ts

Khi 2 người dùng hỏi cùng một câu hỏi với cùng một model, hệ thống không cần gọi AI lần thứ 2. Thay vào đó, nó trả lại kết quả đã lưu trong Redis → tiết kiệm 100% chi phí, phản hồi chỉ 5ms.

📝 Request vào → 🔑 SHA-256(model + messages) → 🔍 Tra Redis → ✅ Trúng? Trả ngay! | ❌ Trượt? Gọi AI
  • TTL (Time To Live): 3600 giây = 1 giờ. Sau 1 giờ, cache tự xóa để đảm bảo kết quả mới.
  • Timeout: Redis GET/SET bị giới hạn 3 giây. Nếu Redis chậm → bỏ qua cache, gọi AI trực tiếp.

💡 Token là gì?

Token là đơn vị đo lường văn bản mà AI xử lý. 1 token ≈ 4 ký tự tiếng Anh, hoặc ≈ 1-2 ký tự tiếng Việt. Ví dụ: "Xin chào bạn" ≈ 5-8 token.

13 Xử lý lỗi thường gặp
❌ "Failed to fetch" / "TypeError: Failed to fetch"

Nguyên nhân: Frontend không kết nối được đến Backend (Gateway chưa chạy, hoặc lỗi CORS)

Cách sửa:

  • Kiểm tra Backend đã chạy chưa: curl http://localhost:8080/health
  • Nếu chưa chạy: npm run dev lại
  • Nếu lỗi CORS: Kiểm tra server.ts → origin: true (không phải '*')
❌ "OpenAI Error: 404 page not found"

Nguyên nhân: Gateway gửi request đến sai URL (ví dụ: gửi cho Ollama thay vì Gemini)

Cách sửa: Mở file chat.ts, tìm dòng tạo adapter. Đảm bảo baseUrl chỉ fallback về Ollama khi providerType === 'OLLAMA':

baseUrl: dbProvider?.baseUrl || (providerType === 'OLLAMA' 
  ? (process.env.OLLAMA_BASE_URL || 'http://127.0.0.1:11434') 
  : undefined)  // undefined = ModelFactory tự dùng URL mặc định
❌ "429 Too Many Requests"

Nguyên nhân: Nhà cung cấp AI (Google, OpenAI...) giới hạn số request/phút

Cách sửa: Chờ 1-2 phút rồi thử lại. Hoặc nâng cấp API Key lên bản trả phí. Hoặc dùng model có quota rộng hơn (gemini-2.0-flash).

❌ "Cannot read properties of undefined (reading 'toFixed')"

Nguyên nhân: Dữ liệu từ API trả về thiếu trường (ví dụ: costs.savings.ollama = undefined)

Cách sửa: Thêm optional chaining: costs?.savings?.ollama?.toFixed(4) || '0.0000'

❌ "Database unavailable" / "Prisma fetch failed"

Nguyên nhân: PostgreSQL chưa chạy hoặc bị crash

Cách sửa: docker compose up -d postgres → Kiểm tra: docker ps

❌ "No models available" trong Playground

Nguyên nhân: Chưa thêm nhà cung cấp AI nào, hoặc API Key sai

Cách sửa: Vào "Nhà cung cấp" → Thêm mới → Chọn loại (GOOGLE_GEMINI) → Dán API Key → Để trống Base URL → Lưu
